## 2026-07-11T06:03:18.554Z

The separating features are: transaction velocity, amount clustering, and counterparty concentration — combined, not alone.
I need to add a customerId field to the transaction schema (it's missing — good catch), since counterparty concentration needs it.

Velocity: count of transactions in a rolling 10-15 minute window, compared against that specific agent's own historical baseline for that time-of-day (z-score on count, not a fixed number — an agent that's normally busy at 5pm shouldn't be judged against a quiet agent's 5pm).
Amount clustering: coefficient of variation (stdev ÷ mean) of amounts within that burst window. Low CV (e.g. below ~0.15) means amounts are suspiciously near-identical.
Counterparty concentration: unique customerIds in the burst window relative to transaction count. Few unique customers making many transactions (e.g. ≤3 customers, 8+ transactions) means repeated requests from the same few accounts.

Scenario B triggers when velocity is high AND (amount clustering is high OR counterparty concentration is high).
The legit demand-spike dataset has high velocity too — but high amount variance (different customers buying different amounts) AND high counterparty diversity (many different real customers). Velocity alone would false-positive on it; the combination shouldn't, because organic demand from many real people doesn't cluster in amount or repeat from the same few accounts the way a synthetic/structured pattern does.
I'm not claiming these thresholds are a universal law of fraud detection — they're a documented design assumption, grounded in the rulebook's own illustrative example (it explicitly describes "much more cash-out than usual," "several near-identical amounts," and "repeated requests from the same accounts" as the signal). I'll state this explicitly as an assumption and expected false-positive boundary in the Data & Assumptions doc, per section 9's requirement — that's honest, not circular, because the combination logic is a real, generalizable pattern (concentrated + clustered + fast beats organic + diverse + fast), even though the exact numeric thresholds are tuned for this dataset.
